§Abstract

Questo documento è il riferimento completo dell'API HTTP della piattaforma Travel AI: copre tutti i domini esposti dal backend Spring Modulith — scoperta pubblica, identità e account, transazioni (prenotazione/pagamento), intelligenza artificiale, viaggi collaborativi, amministrazione e webhook. Ogni dominio elenca i propri endpoint (metodo, percorso, requisito di autenticazione, scopo); i flussi principali includono esempi HTTP di richiesta/risposta. La Sez. 13 fornisce l'indice completo di tutti gli endpoint.

Struttura in stile RFC / OpenID Connect. Prosa in italiano; termini normativi (MUST, SHOULD, MAY) e identificatori (percorsi, header, campi JSON) in inglese.

§Status & convenzioni

Bozza che descrive il comportamento realmente implementato alla data indicata. Le parole chiave in maiuscolo seguono [RFC2119]/[RFC8174]. Nelle tabelle la colonna Auth usa: pub = pubblico (nessun token), bearer = access token richiesto, ADMIN = ruolo ADMIN richiesto. Tutti i percorsi sono relativi al prefisso globale /api.

Base di riferimento

Spring Boot 3.5 / Java 21 / Spring Modulith (40 moduli). Envelope ApiResponse<T>. Token JWT 24h (Bearer). AI: Ollama qwen2.5:7b + nomic-embed-text + pgvector. Rotte e DTO estratti dai controller effettivi.

Parte I — Fondamenti del protocollo

1.Envelope, paginazione, errori, rate limit

1.1Inviluppo di risposta

Normativo. Ogni risposta applicativa MUST essere avvolta in ApiResponse<T> = { success, data, error, meta }; i campi nulli MAY essere omessi. meta = { total, page, limit } per le collezioni paginate.

{ "success": true, "data": { /* ... */ } }
{ "success": true, "data": [ … ], "meta": { "total": 240, "page": 0, "limit": 12 } }
{ "success": false, "error": "Booking not found" }

Figura 1 — Forme canoniche dell'inviluppo

1.2Paginazione

Parametri page (base 0, default 0) e size (default per-endpoint, es. 12). La risposta popola meta. Il client SHOULD richiedere pagine finché il numero di elementi resta pari a limit (scroll infinito).

1.3Errori

Normalizzati in ApiResponse.error da un gestore centrale. Stato HTTP per classe (400/401/403/404/409/429/500). Il campo error MUST NOT divulgare dettagli interni. Registro in Sez. 12.

1.4Rate limit

60 richieste/minuto per IP, finestra scorrevole in Redis → 429 con inviluppo d'errore; fail-open se Redis è irraggiungibile. Il client SHOULD applicare backoff.

2.Autenticazione, ruoli, autorizzazione

Modello ad allow-list: i percorsi pubblici sono elencati esplicitamente; ogni altro percorso richiede un token. Token JWT presentato come Authorization: Bearer <accessToken>, durata 24h; il client web lo tiene in localStorage.ai_access_token. Ruoli: TRAVELER, PARTNER, OPERATIONS, ADMIN. Gli endpoint /api/admin/** MUST richiedere ADMIN.

Normativo — allow-list

Percorsi pubblici: /auth/** (tranne 2fa/setup|enable|disable, resend-verification), /catalog/**, /destinations/**, /attractions/**, GET /forum/**, /reviews/target/** e /reviews/recent, /promo/**, GET /subscriptions/plans, /stats/**, /stories/**, POST /contact, GET /blog|careers|press|help|site-content/**, /actuator/**, /webhooks/**, documentazione. Un nuovo endpoint pubblico MUST essere aggiunto qui, altrimenti risponde 401.

POST /auth/refreshpub{ refreshToken } → nuovo AuthResponse.
POST /auth/logoutpub{ refreshToken } → invalida il refresh token.
POST /auth/social/googlepub{ idToken } → AuthResponse (verifica presso Google).
POST /auth/forgot-passwordpub{ email } → email di reset.

POST /api/auth/login
{ "email":"giulia@example.com", "password":"correct-horse-battery" }

# Caso A — nessun 2FA
→ { "success":true, "data":{ "mfaRequired":false,
      "accessToken":"eyJ…", "refreshToken":"d4f1…", "expiresIn":86400, "role":"TRAVELER" } }

# Caso B — 2FA attivo → challenge, poi /auth/2fa/verify
→ { "success":true, "data":{ "mfaRequired":true, "mfaToken":"mfa_a1b2c3" } }

Figura 4 — Login con e senza secondo fattore

Normativo — 2FA

Con 2FA attivo, il server MUST NOT emettere i token in /auth/login: restituisce solo mfaToken (monouso, breve durata). I token sono emessi solo dopo /auth/2fa/verify. Codice errato → MFA_CODE_INVALID. Il qrDataUri è un'immagine data:image/png;base64,… dell'URI otpauth:// ([RFC6238]).

Figura 5 — Creazione prenotazione multi-componente

Normativo — prezzo server-authoritative

Il server MUST ricalcolare l'importo dai prezzi correnti e dal modello di commissione (10% hotel, 3% altri), applicare sconto Prime ed eventuale riscatto punti, e MUST NOT fidarsi degli importi del client: se divergono, prevale il server. I campi monetari in richiesta sono suggerimenti di visualizzazione.

Errori: annullamento su prenotazione già annullata → BOOKING_ALREADY_CANCELLED; non cancellabile → BOOKING_NOT_CANCELLABLE; id inesistente → BOOKING_NOT_FOUND.

10.Pagamenti — /api/payments

Metodo · PathAuthCorpo → Scopo

L'importo MUST corrispondere al dovuto server-authoritative. La conferma pubblica un evento di dominio che innesca in modo asincrono più conseguenze indipendenti:

POST /payments/{id}/confirm │ ▼ PaymentCompletedEvent ──┬──► BookingPaymentListener → prenotazione = CONFIRMED ├──► LoyaltyPaymentListener → accredito punti fedeltà └──► InvoiceEmailListener → invio fattura PDF via email

Figura 6 — Fan-out asincrono guidato da evento

Il client SHOULD NOT assumere che conferma prenotazione, punti e fattura siano già completati nella risposta sincrona: lo stato aggiornato è osservabile via GET /api/bookings/{id} e nelle notifiche. Errori: PAYMENT_FAILED, PAYMENT_NOT_FOUND.

A.Appendice A — Esempio end-to-end

Non-normativo. Sequenza minima per portare "Giulia" da zero al biglietto:

  3. POST /api/ai/itinerary-plan → piano con hotelId/flightId (§12).
  4. POST /api/bookings con quei riferimenti → PENDING, prezzo ricalcolato (§9).
